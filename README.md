Java SharedValue — Persistent cross-process shared value
=====================================================

A small, zero-dependency Java library that provides a process-safe and thread-safe shared value persisted on disk. The library stores a Serializable value in a binary file and coordinates access across threads and processes using file locks and an in-process read/write lock.

Overview
--------

The `SharedValue<T>` class (see `src/main/java/be/raffaelegigantelli/sharedvalue/SharedValue.java`) lets you persist a Java `Serializable` value in a small binary file located in the system temporary directory (by default inside an application-specific folder). The file is protected with file locks so multiple JVM processes can safely read and write the value. A `ReentrantReadWriteLock` protects concurrent access inside the same JVM.

Key features
------------

- Persists any `Serializable` Java object to disk.
- Cross-process synchronization using `FileLock`.
- Thread-safety within the JVM using `ReentrantReadWriteLock`.
- Simple, single-class API with small dependency surface.

Quick start
-----------

Add this project to your build (local JAR or published artifact). The library is plain Java and has no external runtime dependencies.

Usage example (Java):

```java
import be.raffaelegigantelli.sharedvalue.SharedValue;

// Create or open a shared value named "counter" with default value 0
try (SharedValue<Integer> counter = new SharedValue<>("counter", 0)) {
    // Read the current value (reloads from disk first)
    int current = counter.get();

    // Atomically update the value
    counter.update(v -> v + 1);

    // Or replace and persist a new value
    counter.set(42);
}
```

Constructors
------------

- `SharedValue(String key, T defaultValue)` — create a shared value using the default application folder (auto-detected).
- `SharedValue(String appFolder, String key, T initialValue)` — create a shared value stored under a custom application folder inside the system temporary directory.

Thread and process safety
-------------------------

- Intra-process: protected by a `ReentrantReadWriteLock`.
- Inter-process: protected by `FileLock` obtained on the underlying file channel. Reads take a shared lock; writes take an exclusive lock.

Exceptions
----------

The library uses checked runtime-style custom exceptions to communicate failures:

- `SharedValueInitializationException` — when the storage cannot be initialized.
- `SharedValueLoadException` — when loading fails (IO or deserialization problem).
- `SharedValueSaveException` — when saving fails (IO problem).
- `SharedValueCloseException` — when closing resources fails.

Implementation notes
--------------------

- Values are serialized using Java's standard serialization (`ObjectOutputStream` / `ObjectInputStream`). Ensure your stored type (and its transitive fields) implement `Serializable` and remain compatible across versions if you rely on persisted files.
- Files are stored under `%TEMP%/<appFolder>/<sanitizedKey>.bin` on Windows (or the OS equivalent via `java.io.tmpdir`). Keys and folder names are sanitized to avoid illegal characters.
- The library eagerly creates the storage file and either loads an existing value or writes the provided initial value at construction time.
